# AI 找房助手规则总表

本文档记录当前 MVP 已实现的意图、检索策略、追问规则、排序规则和话术规则，作为后续迭代和客服培训的共同口径。

## 1. 总体原则

- 前端只负责交互、地图和展示，不直接访问 MCP 或模型。
- 后端负责模型意图识别、规则兜底、MCP 查询、排序、话术生成和日志。
- 模型优先理解自然语言；模型失败、低置信度或输出不合法时，后端规则兜底。
- 可执行规则以代码为准，当前主要在 `apps/server/src/assistant.ts`、`apps/server/src/bailianLlmProvider.ts`、`apps/server/src/requirementRules.ts`。
- 客服看到的 chat 回复应是可直接发给客户的话术，系统解释类内容放在右侧摘要和查询结果。

## 2. 意图路由

| 意图 | 典型输入 | 策略 | 是否要求预算/户型 |
| --- | --- | --- | --- |
| `recommend_houses` | `白云东平一室一厅，预算1000左右` | 完整找房推荐，按位置/预算/户型检索和扩圈 | 是 |
| `project_vacancy` | `龙湖31店还有什么空房` | 按项目、门店、楼栋关键词查空房 | 否 |
| `building_detail` | `瑞东石井二期A栋详情`、`某项目有什么户型` | 按项目/楼栋关键词查空房，并汇总户型、价格和可转述介绍 | 否 |
| `area_inventory` | `永泰有什么房子`、`市桥有什么房源` | 按区域、商圈、地铁站关键词查库存概览 | 否 |
| `feature_inventory` | `永泰有没有可养宠物的房子`、`东平有阳台的房源` | 按位置 + 特征关键词查空房，结果需提示以房源详情确认为准 | 否 |
| `move_in_inventory` | `永泰近期可入住的房子`、`本周能入住吗` | 复用特征查询策略，按入住时间关键词查空房 | 否 |
| `payment_inventory` | `永泰有没有押一付一`、`可以月付吗` | 复用特征查询策略，按付款条件关键词查空房 | 否 |
| `commute_ranking` | `永泰到珠江新城上班方便吗` | 解析目的地坐标，按房源到目的地直线距离排序，提醒实际通勤需地图路线确认 | 否 |
| `metro_line_inventory` | `3号线沿途的房源`、`3号线沿线房源` | 遍历已配置线路站点，按站点关键词查沿线库存 | 否 |
| `metro_station_inventory` | `3号线同和站房源`、`同和站附近房源` | 只查具体站点附近，不扫描整条线路 | 否 |
| `area_layout_availability` | `花都狮岭有没有一房` | 按区域 + 户型查空房 | 不要求预算 |
| `price_range` | `白云永泰一居室价格范围` | 按区域 + 户型统计价格范围 | 不要求预算 |
| `distance_ranking` | `白云东平离地铁最近的房源排序` | 按目标点距离排序 | 不要求预算/户型 |

## 3. 位置规则

- `白云区`、`黄埔区` 等行政区太大，不算完整位置，应追问具体地铁站、商圈、街道或小区。
- `嘉禾望岗`、`永泰`、`同和站` 这类具体地铁站或商圈可以作为可查询位置。
- 前端会尝试用高德 POI 解析客户输入，并把可用坐标传给后端。
- 后端会用位置解析结果、内置字典和 MCP 房源坐标做距离排序。
- 房源缺坐标或距离异常时，不应在地图上误展示为靠近需求点。
- 当前 MVP 默认服务广州：前端高德 POI 使用广州 `citylimit`，后端高德解析默认 `AMAP_CITY=广州`。
- MCP 数据层暂未强制按城市字段隔离；推荐类结果会用需求坐标做距离过滤，咨询类关键词查询仍可能依赖 MCP 返回质量。多城市上线前需要增加城市配置、城市字段过滤和跨城异常拦截。

## 4. 地铁规则

- `3号线沿线/沿途` 是线路意图，当前按配置站点逐个查：机场北、机场南、高增、人和、龙归、嘉禾望岗、白云大道北、永泰、同和、京溪南方医院、梅花园、燕塘、林和西、体育西路、石牌桥、岗顶、华师、五山、天河客运站、广州塔、客村、大塘、沥滘、厦滘、大石、汉溪长隆、市桥、番禺广场。
- `3号线同和站房源` 是具体站点意图，只查 `同和`，不扫描整条线路。
- 地铁线路推荐需要标注“靠近哪个站、约多少米/公里”。
- 如果站点坐标解析失败，至少保底标注“靠近某某站”。
- 后续 P1 可补全更多线路和前后站拓扑，而不是手写所有城市数据。

## 5. 追问规则

- 完整找房推荐必须尽量确认：具体位置、预算、户型。
- 如果只缺户型，追问户型，不重复追问位置和预算。
- 如果只缺预算，追问预算，不重复追问位置和户型。
- 如果只有行政区位置，优先追问具体位置。
- 咨询类意图不强制追问预算/户型，例如“永泰有什么房子”“一居室价格范围”“某项目还有什么空房”。
- 客服回复“周边可以/附近也行”时，继承上一轮需求并扩大位置范围。
- 客服明确说“预算可以上浮/贵点也行”时，才扩大预算上限。

## 6. 检索与扩圈

- 严格匹配优先：位置 + 预算 + 户型。
- 无严格匹配时，优先做周边距离和预算策略，不直接扩大到行政区。
- 预算 `1000左右` 默认转为 `800-1200`，类似区间由规则和模型共同解析。
- 查询结果优先使用空置房源 `status=0`。
- MCP 返回坐标时，按距离目标点排序；没有坐标时只能做关键词和基础排序。

## 7. 排序规则

推荐排序综合：

- 空置状态
- 租金是否贴近预算
- 户型是否匹配
- 距离需求点或站点的距离
- 面积
- MCP 返回结果质量

地图模式会隐藏距离异常的房源，并让列表、地图点位和选中卡片保持同一排序。

## 8. 话术规则

- Chat 中助手回复必须是客服可直接发给客户的话术。
- 不应出现“右侧已整理”“我可以继续筛”这类内部系统表达。
- 推荐话术需要说明：有几套、价格范围、前几套具体房源、下一步可如何收窄。
- 没房时要给客户可接受的下一步：周边、前后站、预算或户型调整。
- 地铁站/线路相关话术应标注靠近站点和距离。

## 9. 前端交互规则

- 右侧保留需求摘要和查询/推荐结果。
- 需求摘要展示 AI 理解的字段和置信度状态；客服可手动修正位置、预算、户型和偏好，并按修正结果重新查询。
- 客服话术融合到 chat 回复里，回复气泡右侧提供复制 icon。
- 输入框上方在会话开始时提供快捷输入胶囊；用户发送过一次消息后隐藏。
- 快捷输入只保留示例按钮，不展示额外标题和说明，点击后填入输入框，由客服确认后发送。
- 推荐结果支持列表和地图模式。
- 地图同时展示需求点和推荐房源点位；推荐结果点位可点击查看详情。
- 推荐卡片支持“合适/不合适”反馈；点击“合适”后房源进入“已认可，优先发给客户”分区，点击“不合适”并选择原因后进入“已排除，作为偏好学习依据”分区。
- 已认可/已排除是会话级房源篮子，刷新后应从后端反馈记录恢复，方便客服继续跟进同一个客户。
- 不合适需记录原因，如价格高、位置远、户型不符、没图、装修差、客户不喜欢。

## 10. 反馈与纠错规则

- 客服修正需求后，后端使用修正后的结构化需求重新检索，并把新推荐写回会话。
- 修正位置如果没有坐标，后端仍会按关键词查询；如后续接入更强 POI 解析，应在修正提交时补坐标。
- 房源反馈先做记录，不直接训练模型；当前已用于客户画像、排序权重和话术提示。
- “不合适原因”会把房源移入已排除分区，但不从历史反馈中删除，避免丢失客服判断依据。
- 当前画像维度包括：价格敏感、位置敏感、户型严格、重视图片、重视装修。
- 同类“不合适原因”累计达到 2 次后触发画像标签，例如两次“位置远”会标记客户位置敏感。
- 推荐时会读取画像：位置敏感优先近距离，价格敏感优先预算内，户型严格优先户型一致，重视图片优先有图房源。
- Chat 话术会提示客服当前排序依据，例如“客户之前比较在意位置距离，这次我优先把近的排前面”。
