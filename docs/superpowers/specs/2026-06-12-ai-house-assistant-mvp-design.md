# AI 找房助手 MVP 方案

## 1. 背景与目标

内部客服每天需要根据客户口头需求在房源系统中筛选房源，再组织推荐话术发给客户。当前流程依赖人工经验，容易出现查询慢、放宽条件不一致、推荐理由不清楚、客服话术水平差异大的问题。

本 MVP 的目标是做一个内部客服可用的 AI 找房助手，帮助客服完成从客户需求到房源推荐再到可复制话术的最小闭环。

核心闭环：

```text
客服输入客户需求
-> AI 理解需求
-> 通过 MCP 查询房源
-> 按规则自动扩圈或降级检索
-> 推荐 3-5 套候选房源
-> 生成客服可直接发送的话术
-> 记录查询过程和客服反馈
```

MVP 不追求自动成交，也不替代客服决策。AI 的定位是客服的找房和话术辅助工具。

## 2. MVP 范围

### 2.1 必须支持

- 支持客服用自然语言输入客户需求，例如“帮我找白云东平一室一厅，预算 1000 左右”。
- 自动抽取位置、户型、预算、整租/合租、面积、朝向、入住时间等需求槽位。
- 通过 MCP 只读查询内部房源和楼栋数据。
- 没有严格匹配时，按位置、预算、户型规则逐步放宽。
- 支持最小位置地理化能力：识别常见地铁站、商圈、片区，能基于楼栋坐标做距离排序。
- 推荐 3-5 套候选房源，并说明推荐原因和不完全匹配点。
- 生成客服可直接复制给客户的话术。
- 记录原始需求、结构化需求、MCP 查询参数、推荐房源、话术、客服反馈。
- 前端不暴露 MCP Token 和模型 API Key。

### 2.2 暂不支持

- 不做自动联系客户。
- 不做房源写库、改价、锁房、修改状态。
- 不做复杂 CRM 流程。
- 不做完整 GIS 系统。
- 不做训练平台。
- 不做自动学习客服聊天记录。
- 不做多角色审批流。

## 3. 产品形态

MVP 建议先做独立内部网页，验证价值后再嵌入现有客服后台。

页面结构：

```text
顶部：客户需求摘要
左侧：对话区
右侧：推荐房源卡片
底部：客服话术复制区
```

核心交互：

- 客服输入客户需求。
- AI 返回需求解析和检索过程摘要。
- 系统展示推荐房源卡片。
- 客服可以复制话术。
- 客服可以标记“合适”“不合适”“已复制”“客户要看房”。

房源卡片最小字段：

- 房源 ID
- 楼栋名称
- 房号
- 户型
- 面积
- 租金
- 押金
- 状态
- 位置说明
- 推荐原因
- 风险提示

## 4. 系统架构

```text
内部客服前端
  -> AI 找房助手后端
  -> LLM Provider
  -> Agent Orchestrator
  -> Skill / Policy Engine
  -> MCP Client
  -> house-ai-manager MCP Server
  -> 内部房源数据库
```

职责边界：

- 客服前端：负责聊天、房源卡片、话术复制、反馈采集。
- AI 找房助手后端：负责鉴权、会话、模型调用、MCP 调用、日志、权限和脱敏。
- LLM Provider：封装不同模型供应商，保证模型可替换。
- Agent Orchestrator：负责需求理解、检索计划、工具调用顺序、结果总结。
- Skill / Policy Engine：承载检索策略、位置扩圈、推荐规则和客服话术策略。
- MCP Client：负责以 MCP 协议调用内部房源服务。
- MCP Server：只读暴露内部房源、楼栋、户型统计和详情数据。

前端不得直接访问 MCP Server，也不得持有 MCP Token。

## 5. 模型接入方案

模型层必须做成可替换 Provider，不绑定单一厂商。

统一接口：

```text
LLMProvider
- extractRequirement(input, context)
- generateSearchPlan(requirement, policy)
- generateRecommendation(requirement, houses, searchTrace)
- generateSalesReply(requirement, recommendations, style)
```

可接入模型：

- OpenAI
- 通义千问
- DeepSeek
- 豆包
- 智谱
- 公司内部大模型网关
- 私有化部署模型

MVP 推荐方式：

- 优先使用公司已有大模型网关或国内模型，降低合规与成本压力。
- 模型必须支持稳定中文理解和稳定 JSON 输出。
- 第一版不强依赖模型原生 Function Calling。
- 模型负责理解需求和生成表达，后端代码负责 schema 校验、默认值补全、MCP 调用顺序和安全边界。
- 当前代码采用 `LLMProvider.extractRequirement()` 优先、规则解析兜底的结构；本地 `MockLlmProvider` 先模拟模型理解能力，后续可替换为国内模型或公司模型网关。

推荐执行方式：

```text
模型抽取结构化需求
-> Zod schema 校验
-> 规则补默认值和置信度判断
-> 后端根据规则生成检索计划
-> 后端调用 MCP
-> 后端排序和过滤
-> 模型生成推荐解释和客服话术
```

这样可以降低模型幻觉和错误工具调用风险。

兜底策略：

```text
模型调用失败 -> 使用规则解析
模型输出 JSON 无效 -> 使用规则解析
模型输出置信度低 -> 追问或使用规则补全
模型输出工具名不在白名单 -> 拒绝执行
模型输出房源不存在于 MCP 结果 -> 拒绝展示
```

### 5.1 结构化输出协议

模型输出不得直接进入 MCP 查询。所有模型输出必须经过后端 schema 校验、默认值补全和置信度判断。

需求抽取输出：

```json
{
  "location": {
    "raw": "白云东平",
    "normalized": "东平",
    "city": "广州",
    "district": "白云区",
    "confidence": 0.86
  },
  "budget": {
    "target": 1000,
    "min": 800,
    "max": 1200,
    "confidence": 0.9
  },
  "layout": {
    "bedroom": 1,
    "livingRoom": 1,
    "toilet": null,
    "confidence": 0.95
  },
  "preferences": {
    "rentType": null,
    "direction": null,
    "minArea": null,
    "moveInDate": null
  },
  "missingRequiredSlots": [],
  "shouldAskFollowUp": false,
  "followUpQuestion": null
}
```

检索计划输出：

```json
{
  "steps": [
    {
      "name": "strict_match",
      "tool": "search_houses_geo",
      "arguments": {
        "radiusMeters": 1500,
        "bedroom": 1,
        "livingRoom": 1,
        "minRent": 800,
        "maxRent": 1200,
        "status": 0
      },
      "fallbackReason": null
    }
  ],
  "maxToolCalls": 5,
  "stopWhenResultsAtLeast": 3
}
```

后端校验规则：

```text
JSON 解析失败：重试一次；仍失败则使用保守追问话术
核心槽位缺失：不查房，先追问
置信度低于阈值：标记为 uncertain，并优先追问或使用更宽松但可解释的查询
模型输出工具名不在白名单：拒绝执行
模型输出参数越权：拒绝执行
推荐房源不在 MCP 返回结果中：拒绝展示
```

## 6. MCP 能力设计

### 6.1 现有 MCP 工具

当前已验证可用的只读工具：

```text
search_houses
get_house_detail
search_buildings
get_building_detail
get_house_type_summary
```

这些工具足够支撑关键词式 MVP，但位置需求会比较粗。

### 6.2 MVP 建议补齐

建议新增或修复：

```text
resolve_location
输入自然语言位置，输出标准地点、类型、坐标、置信度和行政区。

search_houses_geo
按坐标、半径、户型、预算、状态查房源，并支持距离排序。

get_house_detail
修复当前依赖 house_images 表导致详情查询失败的问题；如果没有图片表，应返回 images: []。
```

如果短期无法新增 `search_houses_geo`，至少要求楼栋表稳定提供：

```text
building_id
building_name
address
district
street
business_area
lng
lat
nearest_metro_station
nearest_metro_line
distance_to_metro
```

后端可以先自行做距离计算和排序。

### 6.3 数据依赖清单

MVP 数据依赖需要提前确认字段来源和降级策略。

```text
house_id：必需，来自 houses 表；缺失则不能推荐
building_id：必需，来自 houses 表；缺失则只能做房源基础展示
rent_price：必需，来自 houses 表；缺失则不能参与预算匹配
bedroom/living_room/toilet：必需，来自 houses 表；缺失则降低户型匹配分
status：必需，来自 houses 表；默认只推荐空置 status=0
building_name/address：必需，来自 buildings 表；缺失则无法生成可靠位置说明
lng/lat：MVP P0 必需，来自 buildings 表或地理编码补齐；缺失则只能使用关键词匹配并标记风险
district/street/business_area：P0 建议字段，可由地址解析补齐；缺失时使用位置字典兜底
nearest_metro_station/nearest_metro_line：P1 字段；缺失时不做地铁前后站扩圈
house images：可选；缺失时返回 images: []，不能影响详情查询
```

第一版允许用静态位置字典补齐常见地点：

```text
广州常见地铁站
重点商圈
内部高频片区
重点楼栋坐标
```

如果位置字典无法识别客户位置，AI 必须追问，不能猜测坐标。

## 7. 位置理解与扩圈策略

位置需求不能只当关键词处理。比如“白云东平”应先解析为标准地点对象，而不是直接降级成“白云区”。

位置解析结果示例：

```json
{
  "raw": "白云东平",
  "city": "广州",
  "district": "白云区",
  "place": "东平",
  "placeType": "metro_station | business_area | village | road | poi | unknown",
  "center": {
    "lng": 113.0,
    "lat": 23.0
  },
  "confidence": 0.86
}
```

扩圈策略：

```text
地铁站：
本站 1-1.5km -> 同线路前后 1-3 站 -> 低换乘通勤区域 -> 相邻商圈 -> 行政区

商圈/片区：
商圈中心 1.5-2km -> 相邻商圈 -> 同街道/同板块 -> 行政区

小区/楼盘：
小区周边 1km -> 周边 2-3km -> 所属商圈 -> 同街道

行政区：
不直接全区乱搜，优先结合预算和户型找库存集中的板块，并提示客服确认客户是否接受
```

没有严格匹配时，AI 话术必须说明扩圈原因：

```text
东平附近暂时没看到完全匹配的一室一厅 1000 左右房源，我帮您往周边扩大了一圈，优先看距离和预算都比较接近的几套。
```

## 8. Skill 与规则分层

MVP 将规则分成三层：

```text
硬规则：代码或配置
软策略：Skill
表达风格：客服话术 Skill
```

### 8.1 硬规则

硬规则必须稳定、可测试、可审计，放在代码或配置中。

示例：

```text
默认只查空置房源：status=0
1000 左右默认预算区间：800-1200
推荐数量：最多 5 套
MCP 每轮最多调用 5 次
地理扩圈半径：1.5km -> 3km -> 5km
敏感字段不返回给前端
```

### 8.2 MVP Skill

MVP 阶段先保持 Skill 粒度收敛，避免一开始拆出过多模块导致维护成本过高。

MVP 包含：

```text
Search Policy Skill
负责需求抽取、缺失信息判断、位置类型识别、地理扩圈、检索顺序、无结果放宽和推荐理由。

Sales Reply Skill
负责客服话术、销冠沟通结构、语气风格和下一步推进动作。
```

后续当失败案例足够多时，再拆成更细的 `Requirement Skill`、`Location Skill`、`Search Strategy Skill` 和 `Recommendation Skill`。

### 8.3 客服话术 Skill

客服话术 Skill 负责生成客服可直接发送给客户的表达。

它可以蒸馏销冠话术，但不应只复制某个销冠的口头禅。应沉淀沟通结构：

```text
先确认客户需求
再说明已经认真查询
再解释是否有完全匹配
再给出替代方案
再强调匹配点
最后推动下一步动作
```

典型场景：

```text
完全匹配：快速建立信心，推动看房
位置无房：解释原位置暂无，推荐周边替代
预算不足：认可预算，再给守预算和上浮预算两个选择
客户犹豫：降低决策压力，提供对比，不强压
客户需求模糊：温和追问核心条件
```

## 9. 核心业务流程

### 9.1 有足够信息时

```text
1. 客服输入客户需求
2. 模型抽取结构化需求
3. 后端校验核心槽位
4. resolve_location 解析位置
5. 生成严格检索计划
6. 调 MCP 查询房源
7. 无结果则按规则扩圈
8. 对候选房源排序
9. 生成推荐理由
10. 生成客服话术
11. 返回前端
12. 记录日志和反馈
```

### 9.2 缺少核心信息时

缺少位置或预算时优先追问。

示例：

```text
客户只说“一室一厅”：
请问客户主要想看哪个区域或地铁站附近？预算大概多少？

客户只说“东平附近”：
东平附近可以，我再确认下预算和户型要求，大概想看多少价位？
```

### 9.3 无匹配结果时

无结果不能直接说“没有”，也不能直接跳到过大范围。

正确策略：

```text
严格匹配无结果
-> 保持位置，放宽预算
-> 保持预算，扩大距离
-> 找附近地铁站或商圈
-> 给出替代方案
-> 必要时追问客户是否接受
```

## 10. 推荐排序

MVP 排序因子：

```text
位置匹配度
距离目标点远近
户型匹配度
租金贴近预算
是否空置
面积
更新时间
是否有图片或详情
```

每套推荐必须包含：

```text
推荐原因
不完全匹配点
下一步建议
```

示例：

```text
推荐原因：租金刚好 1000，1 室 1 厅，面积 60 平，当前空置。
不完全匹配点：不是东平精确位置，是周边替代。
下一步建议：适合先发给客户确认是否接受周边位置。
```

## 11. 数据与日志

MVP 至少记录：

```text
session_id
客服 ID
客户原始需求
结构化需求
位置解析结果
检索计划
MCP 调用工具
MCP 调用参数
MCP 返回房源 ID
推荐房源 ID
生成话术
客服复制状态
客服反馈
失败原因
创建时间
```

这些日志用于：

- 排查错误推荐。
- 优化扩圈规则。
- 评估模型抽取准确率。
- 蒸馏销冠话术。
- 建立后续评测集。

### 11.1 事件埋点

MVP 至少记录以下事件，便于计算验收指标：

```text
message_sent：客服发送需求
requirement_extracted：模型完成需求抽取
follow_up_asked：AI 发起追问
location_resolved：位置解析完成
mcp_called：后端调用 MCP 工具
mcp_failed：MCP 调用失败
recommendation_shown：推荐房源展示
reply_generated：客服话术生成
reply_copied：客服复制话术
feedback_submitted：客服提交反馈
viewing_intent_marked：客服标记客户有看房意向
```

指标口径：

```text
需求抽取准确率 = 抽取正确样本数 / 标注样本数
推荐采纳率 = 标记合适或客户要看房的推荐会话数 / 展示推荐的会话数
话术复制率 = reply_copied 会话数 / reply_generated 会话数
MCP 调用失败率 = mcp_failed 次数 / mcp_called 次数
无结果解释满意度 = 无结果场景中标记合适的会话数 / 无结果会话数
平均响应时延 = 从 message_sent 到 recommendation_shown 或 follow_up_asked 的耗时
```

## 12. 权限、安全与合规

要求：

```text
MCP Token 只存服务端环境变量
模型 API Key 只存服务端环境变量
前端只调用 AI 找房助手后端
MCP 工具第一版全部只读
客服账号必须鉴权
按客服账号记录查询日志
敏感字段不进入模型上下文
敏感字段不返回前端
限制单次对话 MCP 调用次数
限制单个客服请求频率
```

敏感字段包括：

```text
业主手机号
业主姓名
合同信息
内部成本价
内部备注
财务信息
非公开房源状态
```

## 13. 验收标准

用 30-50 条真实客服需求做灰度测试。

通过标准：

```text
80% 以上请求能正确抽取位置、预算、户型
70% 以上请求能给出可接受房源或合理无房解释
无房时不会直接粗暴扩大到整个行政区
不会编造不存在的房源
推荐结果能追溯到 MCP 返回数据
客服能一键复制话术
MCP Token 和模型 API Key 不暴露到前端
详情接口缺图片时不会导致整体失败
```

核心评估指标：

```text
需求抽取准确率
推荐采纳率
话术复制率
客户继续沟通率
预约看房转化率
无结果解释满意度
平均响应时延
MCP 调用失败率
```

- 记录完整调用日志。

### 第 3 周：客服前端

- 实现聊天界面。
- 实现推荐房源卡片。
- 实现话术复制区。
- 实现反馈按钮。

### 第 4 周：灰度试用

- 形成第一版评测集。

## 15. 关键风险与对策

### 15.1 模型编造房源

对策：

- 模型不能自由编造房源。
- 所有推荐必须来自 MCP 返回结果。
- 前端展示房源 ID。
- 日志记录 MCP 调用和返回。

### 15.2 位置理解过粗

对策：

- 位置先解析为标准地点和坐标。
- 扩圈优先周边和地铁，不直接跳行政区。
- 无法识别位置时让 AI 追问。

### 15.3 国内模型工具调用不稳定

对策：

- 第一版不依赖模型原生 Function Calling。
- 模型输出结构化需求。
- 后端代码控制 MCP 调用。

### 15.4 话术不符合销售习惯

对策：

- 客服话术 Skill 单独维护。
- 从销冠样例中提炼结构。
- 通过客服复制率和反馈迭代。

### 15.5 数据接口不完整

对策：

- 详情接口必须允许图片为空。
- 搜索接口失败时返回可解释错误。
- MCP 工具保持只读和稳定 schema。

## 16. 后续演进

MVP 验证后，可逐步演进：

- 嵌入现有客服后台。
- 支持多轮客户画像。
- 接入地图和地铁站点数据。
- 做销冠话术库和 A/B 测试。
- 基于客服反馈优化排序。
- 支持客户偏好记忆。
- 支持看房预约辅助。
- 支持不同城市和不同项目的独立策略。

## 17. 当前结论

MVP 的核心不是做一个会聊天的机器人，而是建立一个稳定、可追溯、可迭代的找房工作流。

最重要的设计原则：

```text
模型可替换
MCP 标准化
数据只读
位置地理化
规则配置化
话术 Skill 化
推荐可追溯
客服可采纳
```

第一版只要能让客服在真实场景中更快找到合适房源，并生成更专业的话术，就达到了 MVP 目标。
